# v0.1 Open Source PRD — OnCallPilot

> 本 PRD 定义 OnCallPilot 第一个可发布开源版本（v0.1）。v0.1 的产品化承诺**极度收敛**，目标是把"AI 调查 on-call 告警"这条核心叙事用最小可行实现端到端打通。长期完整产品意图见 `docs/superpowers/prds/2026-05-27-product-vision.md`。

## Problem Statement

社区里关心"AI agent 调查 on-call 告警"这件事的工程师，没法在不投入大量基础建设的前提下，亲手验证这个想法是否可行。要么自己从零拼接 LangGraph + Prom + Loki + 一个 webhook，要么去看几个孤立的 blog post，最终大部分人放弃。

OnCallPilot v0.1 要解决的具体问题：**让一个对 on-call 工作流和 AI agent 都好奇的工程师，在一个下午之内能 helm install 起一个端到端的最小可运行系统，发一条 Alertmanager 告警进去，亲眼看到 LangGraph 调用真实 Prom 和 Loki、最终输出一份带证据链的报告。**

不解决的问题（推迟）：GitHub 修复提案、CRD 巡检、Chat 主动查询、事故记忆沉淀、Langfuse trace、多语言、chaos 故障注入、多副本可用性等"完整产品力"问题——这些都在 vision 里，但不在 v0.1 里。

## Solution

v0.1 交付一条单向最短路径：

1. 用户从 Helm Chart 安装 OnCallPilot（api / worker / web 三个 Deployment + DB 迁移 Job），把自己的 Postgres / Redis / Prometheus / Loki / OpenAI 兼容 LLM endpoint 通过统一配置文件传入。
2. Alertmanager 配置一条 webhook receiver 指向 OnCallPilot；或运营者直接用 curl 把一份样例告警 POST 进去。
3. OnCallPilot 立即返回 202 + `incident_id`，把调查任务塞进 Redis 队列；arq worker 拉起，跑一次 LangGraph：
   - 第 1 步固定健康探测（Prom / Loki / LLM endpoint 都通）。
   - 主循环由 LLM 通过 OpenAI 协议 function calling 选工具：Prometheus 错误率 / 延迟 / QPS、Loki 错误日志 / 时间窗日志、Runbook 关键词检索。
   - 每次工具调用前后都落 `tool_calls` 行（参数、结果摘要、状态、耗时）。
   - 工作预算 10 步内收敛，硬上限 20 步兜底。
4. 调查结束输出结构化 final report（summary / suspected_root_cause / impact / evidence / recommended_actions / related_runbook / confidence），写入 `investigation_sessions.final_report`。
5. 用户打开极简 Dashboard：
   - 首页：最近 incident 列表。
   - 点进 incident：看到 alert payload 与关联的 investigation sessions。
   - 点进 investigation：看到工具调用时间线（轮询刷新）+ 最终报告。
6. 整套链路用真实数据源验证，**不内置任何 mock**。

v0.1 的"产品野心"故意压在地板上：能跑、能看、能讲清楚。所有边界扩展留给 v0.2+。

## User Stories

### 部署与上手

1. 作为开源用户，我希望 `git clone` 后看到一份 README，能在 15 分钟内理解 OnCallPilot 是干什么的、需要哪些前置基础设施。
2. 作为开源用户，我希望 README 明确列出 v0.1 需要的外部依赖（Postgres、Redis、Prometheus、Loki、OpenAI 协议 LLM endpoint），并对每一项说"我们不打包，你自己准备"。
3. 作为开源用户，我希望通过 `helm install oncallpilot ./deploy/helm/oncallpilot -f my-values.yaml` 一键部署，不需要自己拼 manifests。
4. 作为开源用户，我希望 chart values 提供一份 `values.example.yaml`，复制改 4-5 个字段就能跑起来。
5. 作为开源用户，我希望敏感信息（DB 密码、LLM API key）走 K8s Secret，YAML 里只有 `*_env: VAR_NAME` 占位，不会把 secret 提交进 git。
6. 作为开源用户，我希望配置文件缺关键字段时进程 fail-fast 立刻报错，不要静默走默认值。
7. 作为开源用户，我希望 `kubectl rollout status` 之后能从 `/healthz` 与 `/readyz` 看到 api、worker、web 三个组件都健康。
8. 作为开源用户，我希望 DB schema 是 Helm pre-install/pre-upgrade hook 自动跑 alembic 迁移，我不必手动 exec 进容器执行命令。

### 告警接入

9. 作为 SRE，我希望把现有 Alertmanager 的 receiver 指到 OnCallPilot `/api/v1/alerts/alertmanager`，立刻能用，不需要改我现有告警规则。
10. 作为 SRE，我希望同一个 alert（同 fingerprint）多次重发只创建一个 incident，不被刷屏。
11. 作为 SRE，我希望 firing → resolved → firing 这样的事件流被识别为多次独立 incident，保留各自 timeline。
12. 作为 SRE，我希望 alert payload 缺关键字段（如 `service` label）时收到 400，告诉我哪个字段缺。
13. 作为开源用户，我希望仓库自带一份样例告警 JSON，用 `curl` 一行命令就能 POST 进去验证。

### 异步调查

14. 作为 SRE，我希望 webhook 立刻返回 202 + `incident_id` + `session_id`，调查在后台跑，不阻塞 Alertmanager。
15. 作为 SRE，我希望调查任务进 Redis 队列由 arq worker 异步消费，worker 重启不丢已经在 DB 里 pending 的 session（v0.1：worker 启动时把 running 超过阈值的 session 标 failed 即可，重试不做）。
16. 作为 SRE，我希望 worker 启动失败 / 不可达时，API 仍然能创建 incident 与 session 行，不会让 webhook 失败。

### LangGraph 与工具

17. 作为 SRE，我希望 AI 调查第 1 步固定先探测基础设施（Prom / Loki / LLM endpoint 都通），如果探测失败立刻收尾给"无法调查"的报告，不浪费步数。
18. 作为 SRE，我希望 AI 主循环用 OpenAI 协议 function calling 自主选工具，能看到它在每一步选了什么工具、为什么（≤30 字 reasoning）。
19. 作为 SRE，我希望 AI 不会反复调用同一个工具的同一组参数（去重护栏）。
20. 作为 SRE，我希望 AI 在 10 步内（可配）收敛；如果失控则 20 步硬上限 abort，仍尽量产出一份"调查未完成"的报告。
21. 作为 SRE，我希望 AI 能在调查中调到 Prometheus 的错误率 / 延迟 / QPS / health 工具，且 PromQL 是基于告警里 `service` label 真实拼出来去查 Prometheus 的。

23. 作为 SRE，我希望 AI 能在调查中调用 runbook 检索，把我团队的 markdown runbook 召回到上下文里参与判断。
24. 作为 SRE，我希望某个工具临时失败（如 Loki 5xx）不会让整个 graph 崩溃，会记一条 "信号缺失" 类型的 evidence 继续走。

### 报告

30. 作为 on-call 工程师，我希望 final report 是结构化 JSON，包含 summary、suspected_root_cause、impact、evidence list、recommended_actions、related_runbook、confidence。
31. 作为 on-call 工程师，我希望 evidence 每条都带 type（metric / log / runbook / signal_missing）、summary、source、confidence。
32. 作为 on-call 工程师，我希望 final report 写入 DB（`investigation_sessions.final_report` JSONB），通过 GET API 拉回。

### Dashboard

33. 作为 on-call 工程师，我希望打开 `http://oncallpilot.example.com` 看到首页：最近 incident 列表，每行展示 alertname、service、severity、status、started_at。

## Implementation Decisions

### 模块边界（v0.1 唯一进入代码库的模块集）

#### 三个进程

- **api**（FastAPI 模块化单体）：HTTP API、入队、读 DB。
- **worker**（arq worker）：从 Redis 队列消费 investigation 任务，执行 LangGraph，写 DB。
- **web**（Next.js App Router，纯前端）：3 个页面 + 轮询的 API client。

不引入 controller / kopf。不引入 chat。

#### 后端核心 deep modules

1. **`ToolRegistry` + `ToolResult` 契约**：所有工具实现同一签名（async callable，统一返回结构）。
2. **`AlertFingerprintDeduper`**：Alertmanager fingerprint 幂等 + firing/resolved 状态机决策的纯函数层，单测覆盖所有分支。
3. **`InvestigationGraph`**：LangGraph 单 agent loop（probe_health → plan → execute_tool → evaluate → continue_or_finish → summarize → persist）。共享 ToolRegistry + LLM client + AuditService。
4. **`AuditService`**：所有工具调用统一落 `tool_calls` 表的窄接口，节点不直接写 DB。
5. **`RunbookStore`**：进程启动时一次性加载所有 markdown runbook 到内存，提供 `search(query, top_k)` 返回 top runbook（含 frontmatter 与正文）。**不依赖 pgvector**，用 token overlap + frontmatter 字段加权的轻量评分。
6. **`PrometheusTool` / `LokiTool` 适配器**：httpx async 客户端 + PromQL/LogQL 拼接，按 `service="<name>"` 单一 label 约定。
7. **`InvestigationService`**：API 侧的 enqueue + Worker 侧的 execute 切分，共享同一份 service 代码（monorepo 内 worker `tool.uv.sources` 指向 api 包）。
8. **`Settings`**（基于 pydantic-settings + 自写 `YamlConfigSettingsSource`）：YAML 加载 + `*_env: VAR_NAME` 占位解析 + fail-fast。

不引入 v0.2+ 才需要的：`Tracer`、`EventBus`、`ServiceCatalogRepository`、`EmbeddingClient`、`ChatGraph`、`HealthCheckResultMapper`、`RemediationService`、`IncidentMemoryService`。这些抽象现在不写，等真正引入对应能力时再加，避免没有具体实现支撑的"空抽象"。

### 数据模型（v0.1 完整 schema）

initial Alembic migration 一次性建以下表，**不含任何 VECTOR 列**：

```
datasource_status(
  id, name, type, status, last_checked_at, latency_ms, message
)

incidents(
  id, alert_fingerprint, alertname, service, severity, summary, description,
  status (open|investigating|resolved|error), source (alertmanager|manual),
  started_at, last_seen_at, resolved_at,
  labels JSONB, annotations JSONB, created_at, updated_at
)
UNIQUE INDEX (alert_fingerprint) WHERE status IN ('open','investigating')
INDEX (service, started_at DESC)

investigation_sessions(
  id, incident_id, source (alert|manual), query,
  status (pending|running|completed|failed),
  final_report JSONB, confidence NUMERIC, step_count, terminated_reason,
  started_at, completed_at
)

tool_calls(
  id, investigation_session_id, tool_name, parameters JSONB,
  result_summary, result_data JSONB, status (success|error),
  latency_ms, error_message, step_index, created_at
)
INDEX (investigation_session_id, step_index)
```

v0.1 显式**不建**的表：`chat_sessions / chat_messages / runbook_documents / runbook_chunks / incident_memories / remediation_actions / service_catalog`。

注意：`tool_calls` 这版**只有 `investigation_session_id`**（不像 vision 那版有跨指 chat 的 CHECK 约束），等 v0.2+ 引入 chat 时再加 `chat_session_id` 列并改约束。这是个明显的、有明确收敛条件的演进点，不构成技术债。

### Final Report Schema（v0.1 简化版）

```json
{
  "summary": "string",
  "suspected_root_cause": "string",
  "impact": "string",
  "confidence": 0.0,
  "evidence": [
    { "type": "metric|log|runbook|signal_missing", "summary": "string", "source": "string", "confidence": 0.0 }
```

### API（v0.1 完整列表）

```
POST   /api/v1/alerts/alertmanager       # 入口，返回 {incident_id, session_id, created}
GET    /api/v1/incidents                 # 列表
GET    /api/v1/incidents/{id}            # 详情 + 关联 sessions
GET    /api/v1/investigations/{id}       # 含 status / tool_calls / final_report
GET    /api/v1/datasources/status        # 数据源健康
POST   /api/v1/datasources/check         # 主动刷新
GET    /healthz                          # liveness
GET    /readyz                           # readiness（检查 DB / Redis / LLM endpoint 可达）
```

v0.1 显式**不提供**：`POST /api/v1/investigations`（手动创建）/ `POST /api/v1/incidents/{id}/close|reopen|investigations`（人工干预）/ chat 相关全部端点 / incident-memories 相关全部端点 / service-catalog 相关全部端点 / `GET /api/v1/investigations/{id}/events`（SSE）。

Incident 状态机自动推进，dashboard 只读：

```
open（webhook firing 新建）
  → investigating（worker pick up，graph 启动）
  → resolved（webhook 收到 status=resolved 推进 / hard cap 不算 resolved）
  → error（worker 异常 / hard cap abort）
```

### LangGraph 节点（v0.1 实际实现的节点集）

```
ingest_alert → probe_health → plan_next_step → execute_tool
                                ↑                    ↓
                                └────── evaluate ────┘
                                            ↓
                                       summarize_report → persist_result
```

- `probe_health`：第 1 步并发跑 Prom / Loki / LLM endpoint 三个健康检查。任一关键源失败 → 跳过主循环直接 summarize（"调查未完成：基础设施 X 不可用"）。
- `plan_next_step`：OpenAI 协议 function calling，工具集 = 当前可用工具家族。强制要求 ≤30 字 reasoning。
- 三条护栏：
  - 同 `(tool_name, hash(args))` 一次 session 内只跑一次。
  - 同工具家族连续 3 次 error → 该家族禁用直到 session 结束。
  - LLM 连续 2 次返回 finish → 真的进 summarize。
- 步数：`max_tool_steps=10` / `hard_step_cap=20`，env 可覆盖。
- `summarize_report`：tools 禁用，强制按 final report schema 输出 JSON。
- `persist_result`：写 `investigation_sessions.final_report / confidence / status='completed'`。

### Tool 清单（v0.1 实际实装的工具）

**Prometheus 家族**：

- `check_prometheus_health`
- `query_prometheus(promql, time?)`
- `get_service_error_rate(service, window='5m')`
- `get_service_latency(service, window='5m', quantile=0.95)`
- `get_service_qps(service, window='5m')`

PromQL 模板按 `service="<name>"` 单一约定，**v0.1 不支持 label 名覆盖**（这是 ServiceCatalog 的能力，留 v0.2+）。

**Loki 家族**：

- `check_loki_health`
- `query_logs_by_service(service, window, limit=100)`
- `query_error_logs(service, window, limit=100)`
- `query_logs_around_time(service, ts, before='2m', after='2m', limit=100)`

LogQL 同样按 `service="<name>"` 单一约定。

**Runbook 家族**：

- `search_runbook(query, top_k=3)`：基于 `RunbookStore` 内存检索，返回 top runbook 的 frontmatter + 正文（必要时按字符数截断到 ~2000 字符塞入上下文）。

**v0.1 显式不实现的工具**：GitHub 全家族、Connectivity 工具（基础设施健康用 Prom/Loki 自身的 health endpoint 覆盖即可）、`summarize_log_patterns`、`search_incident_memory` / `save_incident_memory`。

### Runbook 文档契约（v0.1 与 vision 一致，不弱化）

强 frontmatter + 7 个固定章节顺序（spec §20）。`scripts/seed_runbooks.py` 启动时严格校验，不合规 fail-fast。

差别只在于：v0.1 **不切 chunk、不入库、不算 embedding**，而是 `RunbookStore` 进程启动时把整篇 markdown 加载到内存，每篇 runbook 作为一个检索单元。

### 检索算法（v0.1 不用 pgvector）

`RunbookStore.search(query, top_k)` 评分规则（简单但确定）：

- 把 `query` 与每篇 runbook 的 `(title + frontmatter.services + frontmatter.alertnames + 全文)` 都做小写 + 中文按字切 + 英文按空格切（不引入 jieba 等额外依赖，jieba 留 v0.2+ 真正做向量检索时再决定）。
- Score = 共现 token 数（覆盖 query token 比例）+ `frontmatter.services` 命中加权（×2）+ `frontmatter.alertnames` 命中加权（×2）。
- 按 score 倒序取 top_k，score 全 0 时返回空。

精度不如向量召回，但对 v0.1 内置 5 篇 runbook 来说够用，且**零外部依赖**。

### 配置文件 Schema（v0.1 子集）

```yaml
app:
  log_level: info
  api: { host: 0.0.0.0, port: 8080 }

datasources:
  postgres:   { url_env: POSTGRES_URL }
  redis:      { url_env: REDIS_URL }

  base_url: https://api.openai.com/v1

  max_tool_steps: 10
```

v0.1 显式**不出现**的配置节：`github`、`observability`、`i18n`、`events`、`service_catalog`、`agent.incident_memory`、`agent.chat_max_tool_steps`、`agent.output_language_policy`、`llm.embedding_model`、`llm.embedding_dim`。配置加载器对这些字段不做识别，避免误导用户。

加载机制、`*_env` 占位、fail-fast、单文件全部与 vision 一致。

### 部署形态

Helm Chart `deploy/helm/oncallpilot/`，渲染：

- 3 个 Deployment：api（≥1 replica）、worker（≥1 replica）、web（≥1 replica）。
- ConfigMap（渲染 YAML 配置）。
- 引用 `existingSecret` 注入所有 `*_env` 对应环境变量。
- Services（ClusterIP）。
- Ingress（可选，nginx 默认）。
- Migration Job（Helm hook pre-install/pre-upgrade，执行 `alembic upgrade head`）。
- RBAC：**v0.1 不需要 K8s API 权限**（没有 controller），ServiceAccount 默认无额外 Role。

**不渲染**：CRD、controller Deployment、samples/payment-api（运营者自行决定是否部署，提供独立 manifests 模板供参考）。

### 镜像与包管理

- Python：`uv` 锁定，每个 app 独立 pyproject。
- Web：`pnpm`，单 workspace。
- 每个 app 独立 multi-stage Dockerfile，非 root 运行。

### 国际化处理

v0.1 不引入 next-intl。Dashboard 文案直接中文硬编码，但**所有用户可见字符串集中在 `apps/web/lib/i18n-strings.ts` 单文件**，作为未来抽取到 next-intl 字典的过渡点（不是 over-engineering，只是 1 个文件，目的是让 v0.2+ 引入 i18n 时改动收敛）。

System prompt 仍英文（最佳实践），LLM 用户可见输出语言**v0.1 硬编码中文**（不暴露配置项）。

### 与 vision 文档的"延续点"承诺

v0.1 实施时必须为以下 v0.2+ 演进点预留无返工的空间（设计层面，不写空抽象）：

- `tool_calls.investigation_session_id` 改为支持跨指 chat → 加列 + 改约束的纯 schema 变更。
- 引入 `service_catalog` 表 + ServiceCatalog CRD → 工具适配器从"硬编码 service label"改为"读 service_catalog"的注入点位于 PrometheusTool / LokiTool 的 PromQL/LogQL 拼接函数。
- 引入 `EventBus` + SSE → graph 节点在状态切换处的"将来 publish 事件"位置在 v0.1 已设计好（用 `# TODO(v0.2): publish step.planned event here` 注释占位，但不引入空抽象）。
- 引入 GitHub remediation → `propose_remediation` 节点在 v0.1 不存在，新增节点而非改造现有节点。
- 引入 pgvector + embedding → 替换 `RunbookStore` 实现，签名 `search(query, top_k) -> list[Runbook]` 保持不变。

## Testing Decisions

### 好测试的判定标准

- 只测外部可观察行为：HTTP 响应、DB 落地行、final_report JSON shape、tool_calls 序列。
- 不测 LLM "是不是真聪明"：LLM 行为通过 fake tool registry + 端到端冒烟单测确认"工具被调了、报告被写了、schema 校验通过"即可。
- 同一行为只在最贴近层覆盖一次，不在 unit 与 integration 之间重复造覆盖。
- 测试失败的报告必须指向"哪个对外承诺被破坏"。

### 重点测试模块（v0.1 必须有的测试）

- `AlertFingerprintDeduper`：integration test。覆盖所有 spec §8.1 中的状态机分支（fingerprint 新建 / 命中 open 不创新 / firing-resolved-firing 多 incident / status=resolved 推进等）。这是 v0.1 最容易隐性回归的点。
- `RunbookStore` 解析与校验：unit。frontmatter 缺字段 / 章节顺序不对 / 章节缺失各自 raise。
- `RunbookStore.search` 评分：unit。固定 corpus + 固定 query，断言 top_k 顺序稳定。
- `ToolRegistry` 注册 / 家族禁用 / 去重护栏：unit。
- `PrometheusTool` / `LokiTool`：unit（respx mock）+ integration（真 endpoint，按 `ONCALLPILOT_CONFIG` 提供）。覆盖 PromQL/LogQL 拼接正确性 + 错误响应转 `ToolResult(status=error)` 不抛异常。
- `InvestigationGraph` happy path：integration。fixture-replaced fake registry 返回预设数据，断言：
  - 第 1 步是 probe_health。
  - 至少 1 个 metric 工具 + 1 个 log 工具被调用。
  - `tool_calls` 行全部落地，step_index 递增。
  - `final_report` schema 校验通过，含 suspected_root_cause / evidence / confidence。
  - session 状态 `completed`，`terminated_reason='finished'`。
- `InvestigationGraph` 步数预算：integration。fake registry 让 LLM 一直选工具，断言 hard cap 触发 + `terminated_reason='hard_cap'` + 仍产出 report。
- `InvestigationService.enqueue + worker.execute` 切分：integration。webhook → 入队 → worker 跑完 → DB 状态正确流转。
- `Settings` YAML 加载：unit。env_ref 解析 / 缺 env / 缺字段 / 缺文件各自 fail-fast。
- DB migration upgrade/downgrade：integration（真 Postgres）。
- Helm chart 渲染：CI 跑 `helm template` + `kubectl apply --dry-run=client`。不做 e2e。

### 测试基础设施立场

- 与 vision 一致：**OnCallPilot 不在仓库内打包基础设施**。
- unit 用 respx 等边界 mock，integration 全部依赖真实基础设施（`ONCALLPILOT_CONFIG` 提供），未配置即 skip 并明确打印缺失。
- 不引入 testcontainers / docker-compose / fake exporter。

### Prior Art

v0.1 是这个仓库的第一版代码，**无 prior art**。所有上述测试是为这个 PRD 新建。后续 milestone 应参考 v0.1 的测试组织作为模板（fixture 命名、conftest 结构、integration skip 规约）。

## Out of Scope

以下能力**显式不在 v0.1 范围**（与 user_query 中的 exclusion 列表对齐 + 因边界自然推导出的延伸排除）：

**用户明确排除的（10 项）**：

1. GitHub remediation（issue 创建 / PR proposal）。
2. GitHub MCP。
3. Incident Memory（save / search / review）。
4. ServiceCatalog CRD（含其表、API、控制器同步）。
5. ScheduledHealthCheck（含 HealthCheck CRD 整套）。
6. Chat graph（含 chat_sessions / chat_messages 表、API、前端页）。
7. Langfuse（含 Tracer 抽象 / observability 配置节）。
8. 多语言（含 next-intl 框架接入）。
9. chaos-mesh（含故障注入资源 / 自动化触发）。
10. 多副本 controller / kopf peering（v0.1 根本没 controller 服务）。

**从上述边界自然推导出的延伸排除**：

- SSE + Redis Streams `EventBus`：Dashboard 用轮询。
- `Tracer` 抽象与任何 trace 上报：DB audit 已足够。
- pgvector + embedding 客户端：runbook 用内存关键词检索。
- `verdict / verdict_reason` 字段：服务于 CRD，无 CRD 不需要。
- `related_historical_incidents` 字段：服务于 incident memory，无 memory 不需要。
- `remediation_proposal` 字段与 `remediation_actions` 表：服务于 GitHub remediation。
- `connectivity` 工具家族：基础设施健康用 Prom/Loki 自身 health endpoint 覆盖。
- `summarize_log_patterns` 工具：v0.1 LLM 直接看原始日志窗即可。
- 人工干预端点（close / reopen / re-investigate）：dashboard 只读。
- 手动创建 investigation 端点 `POST /api/v1/investigations`。
- Service label 映射覆盖：强约定 `service="<name>"` 单一 label，运营者必须按此约定打 metric / log label。
- ServiceAccount / RBAC 配置：v0.1 不需要 K8s API 权限。
- `samples/payment-api` 由 Helm chart 渲染：v0.1 不自动部署，仅提供独立 manifests / docker-compose 模板供参考。
- 数据保留 / 自动清理：v0.1 默认不清理 `tool_calls / investigation_sessions`，按 vision PRD 推迟到 operational hardening。
- 镜像发布到 GHCR / 公共 registry：v0.1 不承诺托管镜像，用户自行 build。
- 任何形式的 telemetry 上报到 OnCallPilot 维护方：v0.1 开源版**不内置**任何自动数据上报。
